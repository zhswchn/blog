JS线程

众所周知js是单线程的，js引擎是单线程执行的， 可是浏览器是可以多线程的，js引擎只是浏览器的一个线程而已， 定时器、网络请求、浏览器渲染等等都是由不同的线程去完成的

比如: js引擎遇到setTimeout后会开启一个定时器线程， 然后继续向下执行，定时器线程会在指定时间后往任务队列插入一个任务，等所有的同步任务执行后，再去任务队列中去执行代码

浏览器线程：

  • GUI渲染线程
  • JS引擎线程
  • 定时器触发线程
  • 浏览器事件线程
  • http异步线程
  • 事件循环(Event Loop)轮询处理线程
  • …….

一个浏览器至少实现三个常驻线程：JavaScript引擎线程，GUI渲染线程，浏览器事件触发线程

js线程与进程

一个进程就是一个程序， 比如用浏览器打开１个网页，就是开启了１个进程， 3个网页就是3个进程

一个进程的运行需要，多个线程的配合，比如打开QQ，有接收消息线程，传输文件线程……

浏览器线程有很多，如:

类别A：GUI 渲染线程

类别B：JS 引擎线程

类别C：EventLoop轮询处理线程

类别D：其他线程，有 定时器触发线程 (setTimeout)、http 异步线程、浏览器事件线程 (onclick)等等。

注意: js引擎与GUI引擎是互斥的

js语言设定为js引擎与GUI引擎互斥，也就是就GUI引擎在渲染时会阻塞js引擎计算

因为，如果GUI渲染时js改变了DOM，那么会造成渲染不同步

类别B

js引擎线程， 一般叫主线程， 用来运行js代码(不包括异步代码)

主线程运行JS代码时，会生成个执行栈，可以处理函数的嵌套

执行栈

执行上下文

js所有的代码运行时都在执行上下文中进行的， js中有三种执行上下文:

  • 全局执行上下文， 默认的， 在浏览器中是window对象， this在非严格模式下指向它
  • 函数执行上下文， js函数在调用时会创建一个上下文
  • Eval执行上下文，Eval函数会产生自己的上下文(可忽略)

执行上下文生命周期: 创建 => 执行 => 回收

创建:

当函数调用， 但未执行内部代码之前，会:

  1. 创建变量对象: 初始化函数的参数， 提升函数声明和变量声明
  2. 创建作用域链: 作用域链在变量对象之后创建的
  3. 确定this指向

在一段 JS 脚本执行之前，要先解析代码（所以说 JS 是解释执行的脚本语言），解析的时候会先创建一个全局执行上下文环境，先把代码中即将执行的变量、函数声明都拿出来。变量先暂时赋值为 undefined，函数则先声明好可使用。这一步做完了，然后再开始正式执行程序。

执行:

执行变量赋值，代码执行

回收:

执行上下文出栈等待虚拟机回收

函数声明提升

有两种创建函数的方法:

函数声明: function fn(){}

函数表达式: var fn = function(){}

函数声明是将整个函数提升， 而函数表达式只提升了var f2，所以为undefined

栈是一种数据结构， 具有先进后出的原则， js中的执行栈也是这样的

当js引擎第一次遇到js代码时， 会产生一个全局执行上下文并压入执行栈， 每当遇到一个函数调用，就往执行栈中压入一个新的上下文，执行栈顶部的函数执行完毕后， 会自动出栈， 也就是先进后出

结论:

  • 单线程
  • 同步执行，只有栈顶的上下文处于执行中，其他的需要等待
  • 全局上下文只有一个，在浏览器关闭时出栈
  • 函数的执行上下文没有个数限制
  • 每次调用函数，就会有新的执行上下文为其创建， 调用自身也是如此

**注意:**函数中， 遇到return直接终止可执行代码的执行， 直接出栈

任务队列

一个静态存储结构， 用来存储异步任务成功后的回调函数字符串(未来要执行的回调函数)， 先成功的排在前面，后成功的排在后面

只有异步成功后，才将回调函数放入队列中，比如setTimeout在3秒后执行一个函数，那么该函数在3秒后才进队列的

类别D:

异步: 定时器，http，浏览器事件

主线程执行JS代码时，碰到异步代码，就把它丢给各自相对应的线程去执行:

  • 定时器: 定时器触发线程在接收到代码时就开始计时， 时间到了就将回调函数放进队列
  • http: http异步线程发起http请求，请求成功后将回调函数放进队列
  • 浏览器事件: 浏览器事件先监听DOM， 直到事件触发后，再将回调函数放进队列

也就是说这几个线程主要干两件事：

  1. 执行主线程扔过来的异步代码，并执行代码
  2. 保存回调函数，在未来的某个时刻，通知EventLoop轮询处理线程过来取相应的回调函数然后执行

类别C:

EventLoop轮询处理线程

上面的主线程， 异步线程， 任务队列互相交流需要由EventLoop作为中介去处理

  1. 主线程遇到异步代码， 交给EventLoop去处理
  2. EventLoop再传给异步线程
  3. 异步线程执行成功后把回调函数传回EventLoop
  4. EventLoop把接收到的回调函数放进任务队列里
  5. 主线程同步任务执行完了， 就去任务队列里去看看有没有需要执行的代码

如此往复循环

注意: 只有主线程的同步代码都执行完了，才会去队列里看看还有啥要执行的， 主线程没执行完，队列里的只能等着

所以setTimeout，setInterval的定时，不一定完全按照设想的时间的，因为主线程里的代码可能复杂到执行很久，所以会发生你定时3秒后执行，实际上是3.5秒后执行（主线程花费了0.5秒）

微任务 宏任务

js中的任务分为两种类型:

宏任务: js主线程中的 定时器，事件绑定，网络请求

微任务: promise .then catch finally async await genrator process.nextTick

执行同步代码 ==> 检查微任务并执行 ==> 执行宏任务1 ==> 检查微任务并执行 ==> 执行宏任务2 ==> 检查微任务并执行 ==> 执行宏任务3 …… 循环

宏任务执行完后一定会检查微任务执行:

也就是说之前所有的异步都放进一个任务队列里， 现在分两个任务队列了：宏任务和微任务
